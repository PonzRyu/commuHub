| 項目 | 内容 |
|------|------|
| システム名称 | CommuHub コミュハブ（製品表示名に準ずる） |
| 文書種別 | 要件定義書 |

---

## 1. はじめに

### 1.1 目的

本書は、チーム内予定表における **メンバーの週間日程を一目で把握する** ことを目的とするシステム（CommuHub コミュハブ）について、**利害関係者が合意すべき要件** を定義する。

### 1.2 背景

365 Outlook(Teams)のカレンダーでは複数人所属チームメンバーの方が1つのカレンダー内にまとめて見れるのは可能だが、**個々のメンバーの予定を集約して見ることはできない。**

### 1.3 スコープ（対象）

以下を本システムの対象とする。

- Web ブラウザ（**Google Chrome 上での適切な動作** を満たすこと）および、同一コードベースに基づく **デスクトップ（Electron）** からの利用
- **.icsファイル（iCalendar形式）** の参照・登録・更新・削除
- 登録した.icsファイルに基づいて、週間日程表を生成
- 週間日程のエクスポート(.png)

### 1.4 スコープ外（明示的な非対象）

以下は **現行要件では対象外** とする（必要になった場合は別途要件追加とする）。

- **ユーザー認証・認可・監査ログ** をアプリケーション層で完結する仕様（別途ネットワーク・運用で制御する前提を許容する）
- 週間プランの電子棚札または、電子ペーパーへの連携昨日（今後拡張可能性あり）

## 2. 前提・制約

### 2.1 技術・環境

- **実装言語**: TypeScript を主とする
- **フロント**: Next.js（App Router を採用）
- **UI フレームワーク**: React
- **UI コンポーネント**: shadcn/ui を採用し、Radix UI をベースとしたアクセシブルなコンポーネント設計を行う
- **スタイリング**: Tailwind CSS を使用する
- **ブラウザ**: Google Chrome での動作互換を満たす
- **ランタイム**: Node.js / npm のバージョンはリポジトリの `engines` で **固定** する
- **データベース**: PostgreSQL（開発環境ではコンテナ等を用いる想定に準拠）

### 2.2 カレンダー連携形式

- メールサービスからエクスポート下ICSファイル（.ics）をアプリケーションにアップロードする

### 2.3 配布形態

- **Electron** によるデスクトップ配布が可能であること。デスクトップ利用時も **バックエンド API が別途利用可能** であることを前提とする

## 3. 機能要件

### 3.1 ホーム・ナビゲーション

| ID | 要件 |
|----|------|
| FR-HOME-01 | チーム全員の予定が確認できること |
| FR-HOME-02 | 部署事絞り込みができること |
| FR-HOME-03 | チーム全体の週間日程をリストアップできること |
| FR-HOME-04 | リストアップしたチーム週間日程の項目ごとにメモができること |
| FR-HOME-05 | 表示中の予定を.png画像としてエクスポートできること |
| FR-HOME-06 | ほかのメニューに遷移できること |

### 3.2 部署の管理

| ID | 要件 |
|----|------|
| FR-SEC-01 | 登録済みの部署名を参照できること |
| FR-SEC-02 | 登録済みの部署を更新・削除できること |
| FR-SEC-03 | 部署を新規登録できること |

### 3.3 メンバーの管理

| ID | 要件 |
|----|------|
| FR-MEM-01 | 登録済みの部署名を参照できること |
| FR-MEM-02 | 登録済みの部署を更新・削除できること |
| FR-MEM-03 | メンバーを新規登録できること |
| FR-MEM-04 | メンバーのicsファイルを登録できること |
| FR-MEM-05 | メンバーのicsファイルを更新・削除できること |

## 4. 非機能要件

### 4.1 可用性・運用

| ID | 要件 |
|----|------|
| NFR-OPS-01 | 本番運用における **フロントの配信** と **API の配置** の手順が文書化されていること（`README.md` 等） |
| NFR-OPS-02 | DB スキーマ変更は **マイグレーション** により再現可能な手順で実施できること |

### 4.2 保守性

| ID | 要件 |
|----|------|
| NFR-MAINT-01 | 未使用コードを過剰に蓄積しない方針に従い、変更影響が把握しやすい構成であること（ワークスペース開発ルールに準拠） |
| NFR-MAINT-02 | 可読性のため、必要に応じてコメントを付与すること |
| NFR-MAINT-03 | 共通化が妥当な UI・ロジックは再利用しうる形で整理すること |

### 4.4 セキュリティ（現行の位置づけ）

| ID | 要件 |
|----|------|
| NFR-SEC-01 | 部署・メンバー編集権限はadmin用のパスワードでアクセス可能とすること |
| NFR-SEC-02 | 社内証明書等を扱うクライアント設定について、**リスクを理解した上で** 利用すること（Electron の証明書取扱い等） |

## 5. データ要件（業務視点）

### 5.1 保持すべき情報

- **顧客**: 請求に必要な識別子（会社コード）、名称、請求単価、通貨、請求元表記（SI パートナー名）
- **店舗マスタ**（任意データ）: 会社コード・店舗コードに紐づく店舗名
- **日次利用実績**: 会社・店舗・日付・ラベル合計・商品更新数
- **発行済み請求メタ**: 会社、利用年月、請求書番号、通貨、TTM 等、Excel 生成に必要な付帯情報

### 5.2 整合性

- 同一予定・同一メンバー・同一部署の管理は **二重に矛盾しない** よう、システムが一意性を満たすこと（DB 制約・設計に従う）
